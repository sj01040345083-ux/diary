# 회원가입·로그인 및 사용자 데이터 관리

이 앱은 회원가입과 로그인 기능이 반드시 필요합니다.

관리형 인증 서비스와 데이터베이스를 연동하여 다음 기능을 실제로 구현하세요.

## 인증 기능

* 이메일 회원가입
* 이메일 로그인
* 로그아웃
* 로그인 상태 유지
* 비밀번호 재설정
* 회원가입 및 로그인 오류 안내
* 로그인하지 않은 사용자의 앱 내부 화면 접근 제한
* 로그인 완료 후 Home 화면으로 이동
* 로그아웃 후 로그인 화면으로 이동

회원가입 화면 입력 항목:

* 이름 또는 닉네임
* 이메일
* 비밀번호
* 비밀번호 확인
* 개인정보 처리 안내 동의

비밀번호는 최소 8자 이상으로 검증하고, 비밀번호와 비밀번호 확인 값이 다르면 명확한 오류를 표시하세요.

## 로그인 화면 디자인 (세련되고 따뜻한 감성)

로그인 화면은 단순한 기능 화면이 아니라 사용자가 처음 만나는 중요한 경험입니다.
전체적으로 **깔끔하면서도 따뜻한 감성**을 느낄 수 있도록 디자인하세요.

### 디자인 방향

* 배경: 부드러운 베이지, 크림, 연한 그린 계열 또는 숲 이미지 활용
* 카드형 로그인 박스 중앙 배치 (둥근 모서리, 은은한 그림자)
* 여백을 충분히 사용하여 답답하지 않게 구성
* 버튼은 부드러운 색감 (연한 초록, 따뜻한 브라운 계열)
* 과도한 색상 사용 금지 (2~3가지 메인 컬러 유지)

### 폰트 스타일

* 전체 폰트는 **깔끔하면서 따뜻한 느낌의 산세리프 계열** 사용
* 추천 스타일:
  * 부드러운 둥근 느낌
  * 가독성 높은 중간 굵기
* 제목은 약간 강조, 본문은 편안한 크기 유지
* 줄 간격을 넉넉하게 설정하여 읽기 편하게 구성

### 로그인 화면 구성 요소

* 앱 이름 또는 로고 (상단)
* 간단한 환영 문구 (예: "오늘도 당신의 하루를 기록해보세요")
* 이메일 입력 필드
* 비밀번호 입력 필드
* 로그인 버튼 (가장 강조)
* 회원가입 이동 버튼
* 비밀번호 찾기 링크

### UX 디테일

* 입력 필드 포커스 시 부드러운 색상 변화
* 버튼 클릭 시 자연스러운 애니메이션
* 오류 메시지는 부드러운 색상으로 명확하게 표시
* 로딩 상태 표시 (로그인 중)

## 사용자별 데이터 관리

모든 일기, 감정, 감사, 소비, 수입, 명언 즐겨찾기, 설정 데이터는 로그인한 사용자의 고유 ID와 연결하세요.

다른 사용자의 데이터는 조회·수정·삭제할 수 없도록 데이터베이스 보안 정책을 적용하세요.

각 데이터에는 최소한 다음 값을 포함하세요.

```ts
userId: string;
createdAt: string;
updatedAt: string;
```

로그인한 사용자의 데이터만 불러오고 저장하도록 구현하세요.

로그아웃할 때는 메모리에 남아 있는 사용자 기록과 캐시를 초기화하여 다른 사용자가 이전 사용자의 내용을 볼 수 없게 하세요.

## 로컬 저장과 서버 저장

로컬 저장소는 로그인 사용자의 임시 입력과 오프라인 보조 기능으로만 사용하세요.

최종 데이터는 사용자 계정과 연결된 데이터베이스에 저장하세요.

* 입력 중인 내용 자동 임시 저장
* 서버 저장 성공 후 임시 데이터 정리
* 네트워크 오류 시 사용자에게 안내
* 중복 저장 방지
* 같은 날짜 기록은 새로 추가하지 않고 수정 처리
* 다른 기기에서 로그인해도 기존 기록 조회 가능

## 보호된 화면

다음 화면은 로그인한 사용자만 접근할 수 있도록 보호하세요.

* 홈
* 기록
* 추가
* 리포트
* 설정
* 감사일기
* 명언 즐겨찾기
* 검색

로그인하지 않은 사용자가 보호된 주소에 접근하면 로그인 화면으로 이동시키세요.

## GitHub 보안

인증 서비스 주소, 공개 키 등 환경 설정값은 `.env` 파일로 관리하세요.

```text
.env
.env.local
```

위 파일은 `.gitignore`에 추가하고 GitHub에 올리지 마세요.

예시 환경 변수 파일은 다음처럼 별도로 제공하세요.

```text
.env.example
```

실제 비밀 키, 관리자 키, 서비스 역할 키는 프론트엔드 코드나 공개 GitHub 저장소에 절대 포함하지 마세요.

기존 GitHub 코드에 키가 이미 노출된 경우 단순히 파일만 삭제하지 말고 해당 키를 즉시 폐기하거나 재발급하세요.

## 기존 프로젝트 처리

기존 GitHub 저장소와 파일을 즉시 모두 삭제하지 마세요.

작업 시작 전에 현재 상태를 백업 브랜치로 보존하세요.

예:

```bash
git checkout -b backup-old-version
git push -u origin backup-old-version
```

그 후 메인 개발 브랜치에서 기존 프로젝트 구조를 확인하세요.

기존 `diary.html`이 새 React 프로젝트에서 사용되지 않는다면 다음 중 하나로 처리하세요.

1. `legacy/diary.html`로 이동하여 임시 보관
2. 백업 브랜치에만 보존하고 메인 브랜치에서는 삭제

다음 파일은 삭제하지 말고 새 프로젝트의 적절한 위치로 이동하세요.

* 숲 배경 이미지
* 명언 Markdown 파일
* 기존 디자인 자료
* 필요한 문구와 데이터

기존 코드를 무조건 모두 삭제하지 말고, 재사용 가능한 파일과 삭제 가능한 파일을 먼저 구분한 후 작업하세요.
